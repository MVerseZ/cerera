# Pallada Virtual Machine

Pallada — это виртуальная машина для выполнения смарт-контрактов в блокчейне Cerera. VM спроектирована с использованием принципов чистого кода и SOLID, обеспечивая высокую производительность и надежность выполнения контрактов.

## 🎯 Обзор

Pallada VM — это стековая виртуальная машина, аналогичная EVM (Ethereum Virtual Machine), но оптимизированная для блокчейна Cerera. VM выполняет байткод контрактов, управляет состоянием, памятью и газом.

### Основные возможности

- ✅ **Стековая архитектура** — операнды хранятся в стеке (256-битные числа)
- ✅ **Линейная память** — динамически расширяемая память для выполнения
- ✅ **Persistent Storage** — постоянное хранилище для данных контракта
- ✅ **Система газа** — контроль потребления ресурсов
- ✅ **Межконтрактные вызовы** — поддержка вызовов между контрактами
- ✅ **Обработка ошибок** — RETURN и REVERT для управления выполнением

## 🔧 Основные компоненты

### 1. Stack (Стек операндов)

Стек для хранения 256-битных значений (`*big.Int`). Максимальная глубина — 1024 элемента.

**Операции:**
- `Push(value *big.Int)` — добавить значение на стек
- `Pop() (*big.Int, error)` — извлечь значение со стека
- `PeekAt(depth int) (*big.Int, error)` — посмотреть значение на глубине
- `Swap(n int)` — обменять элементы
- `Dup(n int)` — дублировать элемент

**Оптимизация:** Переиспользование элементов слайса для снижения аллокаций памяти.

### 2. Memory (Память выполнения)

Линейная байт-адресуемая память, расширяемая динамически до 1 MB.

**Операции:**
- `Store(offset *big.Int, value *big.Int)` — сохранить 32-байтное слово
- `Load(offset *big.Int) (*big.Int, error)` — загрузить 32-байтное слово
- `StoreByte(offset *big.Int, value byte)` — сохранить байт
- `Copy(offset, length *big.Int) ([]byte, error)` — копировать данные
- `Set(offset *big.Int, data []byte)` — установить данные

**Оптимизация:** Начальная емкость 1KB для снижения переаллокаций.

### 3. GasMeter (Счетчик газа)

Управление потреблением газа во время выполнения.

**Методы:**
- `ConsumeGas(amount uint64, reason string) error` — потреблять газ
- `GasRemaining() uint64` — оставшийся газ
- `GasUsed() uint64` — использованный газ
- `GasLimit() uint64` — лимит газа

### 4. Context (Контекст выполнения)

Содержит информацию об окружении выполнения контракта:

```go
type Context struct {
    Caller    types.Address    // Адрес вызывающего
    Address   types.Address    // Адрес контракта
    Value     *big.Int         // Переданное значение
    Input     []byte           // Входные данные
    GasLimit  uint64           // Лимит газа
    GasPrice  *big.Int         // Цена газа
    BlockInfo *BlockInfo       // Информация о блоке
    Storage   StorageInterface // Интерфейс storage
    Call      CallInterface    // Интерфейс вызовов
}
```

### 5. VM (Виртуальная машина)

Основной компонент, управляющий выполнением байткода.

**Методы:**
- `NewVM(code []byte, ctx *Context) *VM` — создать VM
- `Run() ([]byte, error)` — выполнить байткод
- `GasUsed() uint64` — получить использованный газ

## 🎨 Принципы проектирования

### SOLID Principles

1. **Single Responsibility Principle (SRP)**
   - `Stack` — только управление стеком
   - `Memory` — только управление памятью
   - `GasMeter` — только управление газом
   - `VM` — только управление выполнением

2. **Interface Segregation Principle (ISP)**
   - `StorageInterface` — отдельный интерфейс для storage
   - `CallInterface` — отдельный интерфейс для вызовов
   - `GasMeter` — интерфейс для управления газом

3. **Dependency Inversion Principle (DIP)**
   - Зависимость от абстракций (`StorageInterface`, `CallInterface`, `GasMeter`)
   - Легкая замена реализаций для тестирования

4. **Strategy Pattern**
   - Каждый опкод имеет свою функцию-обработчик
   - Легкое добавление новых опкодов

## 💻 Использование

### Базовый пример

```go
package main

import (
    "math/big"
    "github.com/cerera/internal/pallada"
    "github.com/cerera/internal/cerera/types"
)

func main() {
    // Байткод: PUSH1 42, PUSH1 0, MSTORE, STOP
    code := []byte{
        0x60, 0x2a, // PUSH1 42
        0x60, 0x00, // PUSH1 0
        0x52,       // MSTORE
        0x00,       // STOP
    }

    // Создаем контекст
    ctx := pallada.NewContextWithStorage(
        types.Address{},  // caller
        types.Address{},  // address
        big.NewInt(0),    // value
        nil,              // input
        10000,            // gasLimit
        big.NewInt(1),    // gasPrice
        &pallada.BlockInfo{
            Number:    1,
            Timestamp: 1234567890,
            Hash:      make([]byte, 32),
        },
        newMockStorage(), // storage
    )

    // Создаем и запускаем VM
    vm := pallada.NewVM(code, ctx)
    result, err := vm.Run()
    
    if err != nil {
        panic(err)
    }
    
    // Проверяем результат
    gasUsed := vm.GasUsed()
    fmt.Printf("Gas used: %d\n", gasUsed)
}
```

## ⛽ Система газа

### Базовые стоимости

```go
const (
    GasZero        uint64 = 0     // Бесплатные операции
    GasBase        uint64 = 2     // Базовая стоимость инструкции
    GasVeryLow     uint64 = 3     // Очень дешевые операции
    GasLow         uint64 = 5     // Дешевые операции
    GasMid         uint64 = 8     // Средние операции
    GasHigh        uint64 = 10    // Дорогие операции
    GasSLoad       uint64 = 100   // Загрузка из storage
    GasSStore      uint64 = 200   // Сохранение в storage (базовая)
    GasSStoreSet   uint64 = 20000 // Сохранение нового значения
    GasSStoreReset uint64 = 5000  // Изменение существующего значения
    GasCall        uint64 = 100   // Базовый CALL
    GasCallValue   uint64 = 9000  // Дополнительно при передаче value
)
```

### Расчет газа для памяти

Расширение памяти требует дополнительного газа:
- Формула: `(expansion² / 512) + (expansion * 4)`
- Где `expansion` — количество новых байтов

### Расчет газа для SSTORE

- Новое значение (0 → non-zero): 20000
- Изменение значения (non-zero → non-zero): 5000
- Удаление значения (non-zero → 0): 200

## ⚡ Производительность

### Оптимизации

1. **Переиспользование памяти**
   - Стек переиспользует элементы слайса
   - Память не обрезается после очистки

2. **Начальные емкости**
   - Стек: 16 элементов
   - Память: 1KB

3. **Эффективные операции**
   - Использование `big.Int` для 256-битных чисел
   - Минимум аллокаций

### Ограничения

- **Максимальная глубина стека:** 1024 элемента
- **Максимальный размер памяти:** 1 MB
- **Размер слова:** 32 байта (256 бит)
